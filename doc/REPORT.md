# Отчет по домашней работе по изучению разных уровней оптимизации инференса

## Базовый инференс

### Описание архитектуры
Для домашнего задания была реализована следующая архитектура:
- Сервис `app`: основной сервис с моделью и ручкой для расчета эмбеддинга по входному тексту
- Сервис `bench`: сервис для проведения бенчмарка (для этого в нем есть специальная ручка). Сервис поднимает дополнительным процессом locust для создания нагрузки (причем параметры нагрузки подаются через ручку). Затем в процессе нагрузки собираются метрики по нагрузке на cpu и gpu, а также метрики от locust: latency (p50, p95, p99), throughput_rps.

### Используемые метрики
- `latency (p50, p95, p99)`

Это основные необходимые метрики для оценки скорости обработки запросов сервисом. p50 характеризует типичное время ответа для большинства пользователей, а p95 и p99 позволяют выявить хвостовые задержки, которые могут указывать на проблемы под нагрузкой.

- `throughput_rps` 

Число запросов, обрабатываемых сервисом в секунду напрямую характеризует пропускную способность сервиса, что является важным показателем для нашего сервиса. По нему мы можем оценить способности сервиса.

- `cpu_avg_pct` и `cpu_max_pct`

Средняя и пиковая загрузка CPU. Помогает понять, является ли сервис CPU-bound, и оценить запас по ресурсам для горизонтального или вертикального масштабирования.

- `mem_avg_mb` и `mem_max_mb`

Среднее и пиковое потребление оперативной памяти. Позволяет отслеживать утечки памяти при длительной нагрузке и оценить базовое потребление модели для планирования ресурсов.

## Конвертация модели в ONNX
Для этого был обновлен сервис: теперь из окружения подтягивается переменная (задается в docker-compose), отвечающая за выбор бэка модели. Также в scripts лежит скрипт конвертации модели в ONNX. 
### Результаты бенчмарка

**Параметры нагрузки:**

| Параметр | Значение |
|----------|----------|
| Количество пользователей | 30 |
| Задержка перед запросами | 0.5с |
| Длительность | 60 с |

**Результаты:**

| Метрика | Значение |
|---------|----------|
| **Throughput** | 57.14 RPS |
| **Latency p50** | 22 мс |
| **Latency p95** | 64 мс |
| **Latency p99** | 99 мс |
| **Всего запросов** | 3 369 |
| **Ошибки** | 0 |
| **CPU avg** | 290.75% |
| **CPU max** | 412.4% |
| **RAM avg** | 494.15 МБ |
| **RAM max** | 496.46 МБ |

Видим очень приятный результат, ONNX очень сильно выигрывает torch инференс: медианная задержка упала с 76 до 22 мс - в 3.5 раза, p95 сократился с 280 до 64 мс (в 4.4 раза), p99 — с 380 до 99 мс (в 3.8 раза). Throughput вырос на ~15%. 

Потребление CPU выросло. Это может быть связано с тем, что ONNX интенсивнее нагружает процессор, ускоряя инференс.

Также потребление памяти снизилось на ~11%, так как ONNX более оптимальный по памяти, чем torch.

## Динамическое батчирование
Для этого была добавлена отдельная ручка для асинхронного получения эмбендинга, а также аналогично с ONNX из среды берется булевая переменная использования динамического батчирования. Батчирование работает следующим образом: приходит первый запрос, в течении `BATCH_MAX_WAIT_MS = 1` ждем другие запросы, затем уже полученный батч отправляем в модель.
